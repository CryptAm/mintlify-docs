---
title: Оптимизация онбординга
description: Уменьшите трение с помощью потоков без обязательного подключения кошелька и четких демонстраций ценности
---

Оптимизируйте процесс онбординга, чтобы повысить вовлеченность и удержание пользователей. Это руководство описывает лучшие практики, которые удержат ваших пользователей в приложении и вовлеченными.

### Обзор

Демонстрируйте ценность мгновенно и избегайте блокирующих действий.

- Сделайте первое взаимодействие мгновенным и неблокирующим
- Аутентифицируйте только тогда, когда это требуется для целей безопасности, и откладывайте запросы до необходимости
- Предпочитайте встроенный Base Account; предлагайте подключение/смену кошелька только для альтернативных вариантов, никогда не блокируя доступ
- Используйте постепенное раскрытие информации, связанное с намерением (купить, опубликовать, персонализировать)
- Удерживайте пользователей в приложении с помощью [дейтсвий SDK для ссылок](/mini-apps/features/links); избегайте ненадежных статических URL

### Рекомендуемый поток онбординга

<Steps>
<Step title="Первая отрисовка">
- Покажите немедленную ценность (демо-контент, пример состояния или режим только для чтения)
- Персонализируйте мгновенно с помощью [`context`](/onchainkit/latest/components/minikit/provider-and-initialization) профиля пользователя
- Покажите один четкий призыв к действию (CTA), который ведет к значимому действию (например, "Опубликовать сообщение", "Купить токен", "Подписаться на пользователя")
</Step>

<Step title="Пользователь инициирует защищенное действие">
- Запускайте Sign In with Farcaster (SIWF)/Quick Auth только тогда, когда это необходимо, согласно [Аутентификации](/mini-apps/features/Authentication)
- Для ончейн-действий автоматически используйте Base Account. Устраните явные процессы подключения кошелька
- Альтернативные кошельки: предлагайте неблокирующую опцию подключения/смены без ограничения доступа к исследованию
</Step>

<Step title="Отпразднуйте и усильте">
- После успеха предложите социальные действия через [действия SDK](/mini-apps/features/links) и [Шэринга и социальный граф](/mini-apps/features/sharing-and-social-graph)
- Предложите следующий шаг: сохранить, подписаться или поделиться - оптимизируйте с помощью [Поиска и обнаружения](/mini-apps/troubleshooting/how-search-works)
</Step>
</Steps>

### Работающие UX-паттерны

<Tip>
- Постепенные запросы: спрашивайте только когда нужно (покупка, публикация, персонализация)
- Четкий текст: объясняйте, почему вы спрашиваете ("Войдите, чтобы сохранить свой счет")
</Tip>

- Одноразовый диплинк (пользователи Connect Account): если SIWF требует одноразовую ссылку на Farcaster для регистрации адреса аутентификации, представьте это как быструю, одноразовую настройку и плавно верните пользователя обратно
- Дружественные запасные варианты: если аутентификация пропущена или провалилась, разрешите продолжать просмотр в режиме только для чтения

### Рекомендации по аутентификации и кошелькам

#### Аутентификация

- Только когда вашему бэкенду нужен проверенный пользователь
- Используйте SIWF/Quick Auth для создания сессии (JWT) при необходимости

<Warning>
Не используйте контекст мини-приложения как основную аутентификацию (его можно подделать)
</Warning>

Подробнее в [Аутентификации](/mini-apps/features/Authentication).

#### Кошельки
- Приложение Base предоставляет встроенный Base Account. Это должен быть кошелек по умолчанию, используемый вашим приложением для упрощения взаимодействий.
- Не показывайте кнопку подключения при первой загрузке
- Если вы поддерживаете другие кошельки, показывайте подключение/смену как опциональную и неблокирующую
- Используйте компонент Wallet из OnchainKit или хуки Wagmi по необходимости

### Не используйте сырые диплинки

<Warning>
- Всегда используйте официальные действия SDK для кросс-клиентной совместимости (например, создание каста, просмотр профиля)
- Это предотвращает тупиковые ситуации и обеспечивает согласованное поведение на разных платформах
</Warning>

Узнайте, как их реализовать с помощью [действий SDK](/mini-apps/features/links).

### Измеряйте активацию и итерируйте

<Info>
- Определите активацию как первое успешное защищенное действие (например, первый пост, первое ончейн-действие)
- Отслеживайте воронку: первая отрисовка → клик по намерению → запрос аутентификации/кошелька → успех → шаринг/сохранение
</Info>

- Анализируйте поведение пользователей с аккаунтом Create Account и Connect Account, чтобы выявить точки трения
- Смотрите: Base Build Analytics (скоро)

### Контрольный список реализации

<Check>
- Стартовый экран можно использовать без запросов на аутентификацию или подключение кошелька
- Запускайте SIWF/Quick Auth только тогда, когда это нужно бэкенду
</Check>

- Используйте контекст MiniKit только для аналитики; избегайте его использования как основной аутентификации
- Бесшовно используйте Base Account для ончейн-действий; без предварительного процесса подключения
- Если поддерживаете альтернативные кошельки, предоставляйте опциональное, неблокирующее подключение/смену
- Используйте действия SDK для социальных потоков (создание/просмотр) вместо диплинков
- Предоставляйте шаг расшаривания после успеха и следующее четкое действие
- Тестируйте пути как для Create Account, так и для Connect Account
